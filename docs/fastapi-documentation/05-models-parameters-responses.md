# Models, parameters, and responses

## Pydantic models and `Field`

On request and response models, use Pydantic v2 **`Field`** for:

- **`description`**: text shown in the OpenAPI model schema
- **`examples`** / **`json_schema_extra`**: example payloads in Swagger/ReDoc
- **Constraints** (`ge`, `max_length`, …): appear as validation rules in the schema

Path and query parameters can also set `title`, `description`, and `example` via `Query`, `Path`, `Header`, etc.

## `response_model`

**`response_model=YourModel`**: filters/validates the response, documents the success body, and (with appropriate config) can hide internal fields. Pair with a clear **`response_description`** on the path operation for clarity.

## Non-JSON and custom classes

**`response_class=…`** (e.g. `Response`, `HTMLResponse`, file streaming) changes how the response is sent. The OpenAPI description of `content` types should match what you return (`text/csv`, `application/geo+json`, …). If the autogenerated schema is too vague, add **`responses={…}`** with explicit `content` and examples.

## Additional status codes: `responses={...}`

Document errors and alternate success codes explicitly, for example:

- **400** / **404** for business rule or “not found”
- **401** / **403** for auth failures
- **422** is often already present for validation

Use the **`responses`** parameter on the path operation (and/or `HTTPException` detail patterns) so the spec lists more than a single 200. See FastAPI [Additional Responses](https://fastapi.tiangolo.com/advanced/additional-responses/).
